# 设计文档修正执行清单

## 1. 文档用途

本文档不是讨论记录，而是后续修复线程执行设计文档修正时必须遵守的指令清单。

目标：

1. 把已经确认的修正决策写成稳定、可执行、无歧义的指令。
2. 让后续修复线程不需要再自行判断方向，只需要按本清单落地。
3. 避免不同线程对同一术语、同一能力边界、同一命名映射产生不同理解。

## 2. 执行边界

后续修复线程必须遵守以下边界：

1. 本文档中的决策视为已确认结论，不再二次发明方案。
2. 若后续修复线程需要修改其他设计文档，应以本文档为唯一修正口径。
3. 未在本文档中明确允许的新增概念、字段、兼容层、保留语义，一律不要自行补充。
4. 若发现旧文档内容与本文档冲突，以本文档为准。
5. 若后续修复中出现新的重大分歧，应先回到本文档补充决策，再继续修复其他文档。
6. 不允许把“内部自动补齐”“内部自愈”“读前修复”写入 GET 接口语义。

## 3. 已确认的硬性决策

以下内容已经确认，后续线程不得擅自修改方向：

1. `link` 不再作为 v1 的可写绑定类型。
2. `link` 不再作为 v1 的运行时展示语义保留。
3. v1 中所有与 `link` 相关的接口说明、DTO 说明、错误语义、巡检项、兼容性描述，统一从修订范围中删除。
4. 若后续确实出现明确使用场景，再在后续版本中重新引入，并重新定义完整契约。
5. 不接受“`summaries` 在 v1 固定映射为 `insight_records.record_type = 'summary'`”这一方案。
6. 与 `summaries` 对应的底层表名统一从 `insight_records` 修正为 `summary_records`。
7. `record_type` 字段从该表及其相关契约中移除。
8. 不再保留“总结 / 认知”混合建模。
9. 与该能力相关的术语统一收敛为 `summary` / `summaries` / `summary_records`。
10. 不再继续使用 `InsightRecord`、`insight_records`、`insights.csv`、`record_type` 这一套命名。
11. GET 接口严格保持只读语义。
12. 读就是读，不允许补齐，不允许顺手修复，不允许以隐式修复掩盖错误。
13. 缺失就是缺失，错误就是错误，必须返回明确错误语义，不能混淆、不能模糊。
14. `ProjectViewport` 不采用“默认值返回”方案。
15. 若不存在 `ProjectViewport` 记录，则必须返回 `PROJECT_VIEWPORT_NOT_FOUND`。
16. 后端不负责为缺失的 `ProjectViewport` 伪造默认对象。
17. 前端若需要初始显示 `(0,0,1)`，由前端本地 UI 自行决定，不通过接口伪装为已存在资源。
18. `Prompt` 删除不采用“阻止删除并返回引用清单”方案。
19. `Prompt` 删除采用“同事务清理所有相关绑定”的方案。
20. 凡是一次操作同时涉及多个数据源，必须通过统一事务抽象执行。
21. 不允许在不同业务点各自发明一套跨数据源事务处理方式。
22. 后续修复文档时，必须把“统一事务抽象”写成全局基础能力，而不是单个接口的临时说明。
23. `workflow_node_actions` 更新时，若 `actionType` 发生变化，则必须同时提交配套 `targetRef`。
24. `workflow_node_actions` 更新校验必须基于“更新后的最终有效状态”，而不是分别校验单字段。
25. 若 `actionType` 变化但缺少配套 `targetRef`，统一返回 `422 + VALIDATION_ERROR`。
26. `Project` 删除时，必须同步移除全部方案绑定（`solution_projects`），但不删除任何 `Solution` 实体。
27. `GET /api/projects/{projectId}/solutions` 默认排序改为按 `updatedAt`，`sortOrder` 仅作为附带信息返回。
28. `ProjectNode` 结构不变量为：每个节点必须且只对应一条 `project_node_relations` 记录；根节点也必须存在一条 `parent_project_node_id IS NULL` 的关系记录。
29. `GET /api/projects/{projectId}/nodes` 的 `parentNodeId` 采用三态语义：未传返回全量扁平列表；`null`/空值返回根层节点；传具体节点 ID 返回其直接子节点；默认排序仅在当前返回集合内生效。

## 4. 统一替换规则

后续凡涉及 `summaries` 能力，统一按以下规则修正：

1. 表名：`insight_records` -> `summary_records`
2. 领域对象名：`InsightRecord` -> `SummaryRecord`
3. CSV 文件名：`insights.csv` -> `summaries.csv`
4. 字段处理：移除 `record_type`
5. 目录能力表述：`总结 / 认知目录` -> `总结目录`
6. 导入导出表述：所有 `insights` 相关描述统一替换为 `summaries`
7. DTO、接口返回、错误说明、巡检描述中，凡出现 `insight` 语义，统一改为 `summary`

后续凡涉及工具配置真实键名，统一按以下规则修正：

1. 配置真实键名统一使用 `tool_key`
2. 不再使用 `toolKey` 指代配置中的真实键名
3. 若文档是在表达“配置项引用目标”，必须写 `tool_key`
4. 若未来出现 HTTP DTO 字段，才允许单独讨论是否映射为 `toolKey`

### 4.1 已确认的统一术语规则：`tool_key`

以下规则已经确认，后续线程直接执行，不再讨论：

1. 第 9 章中凡是表达 `tool` 绑定目标指向本地工具配置键名的地方，统一改为 `tool_key`
2. 第 16 章中凡是表达运行时工具目标引用的地方，统一改为 `tool_key`
3. 第 17 章中凡是表达工具目标校验或工具目标缺失的地方，统一以 `tool_key` 为准
4. 不允许在同一份文档里同时并存“配置真实键名是 `tool_key`”和“配置真实键名是 `toolKey`”两种写法
5. 后续修复时，若旧文档出现 `toolKey`，应判断其语义：
   - 如果表达的是配置真实键名，必须直接改为 `tool_key`
   - 如果表达不清或存在歧义，直接去歧义，不保留双写法，不保留兼容说明，不保留括号补充写法

### 4.2 已确认的顺序字段规则：`sortOrder`

以下规则已经确认，后续线程直接执行，不再讨论：

1. `sortOrder` 只是局部顺序字段，不是全局排序字段。
2. `sortOrder` 只在同一父级范围内或同一方案范围内有意义。
3. 禁止把 `sortOrder` 当作跨父级、跨方案、跨集合的通用排序依据。
4. `ProjectNode` 本体承载的是节点自身功能和业务信息，不承载局部顺序语义。
5. 节点的顺序信息属于结构关系或展示关系，不属于节点本体字段语义。
6. 后续修复时，凡是把 `sortOrder` 直接写成 `ProjectNode` 本体稳定属性的地方，都必须重新表述其来源是关系层，而不是节点本体层。
7. 后续修复时，凡是使用 `sortOrder` 进行跨集合默认排序的地方，都必须改掉。

### 4.3 已确认的跨数据源事务规则

以下规则已经确认，后续线程直接执行，不再讨论：

1. 当前系统存在多个数据源参与同一业务操作，后续文档修正必须显式承认这一事实。
2. 只要一次操作同时修改两个或以上数据源，就必须走统一事务抽象。
3. 这个统一事务抽象是全局基础能力，不是某一个接口私有的实现技巧。
4. 后续文档中凡是涉及“数据库记录 + 文件系统内容”“结构化数据 + 另一数据源”同时修改的场景，都必须引用同一套事务边界语义。
5. 不允许把这类操作写成“先改 A，再尽量改 B”的松散流程。
6. 不允许每个模块、每个接口各自定义不同的补偿逻辑名称或事务口径。
7. 若某一操作无法在统一事务抽象下保证一致性，则文档必须把失败语义写清，不能模糊处理。

### 4.4 已确认的 `Prompt` 删除规则

以下规则已经确认，后续线程直接执行，不再讨论：

1. `Prompt` 删除时，若存在引用该 Prompt 的相关绑定，不阻止删除。
2. `Prompt` 删除时，必须在同一事务边界内同步清理所有相关绑定。
3. 这里的“相关绑定”至少包括所有引用该 Prompt 的 `workflow_node_actions`。
4. 不允许把 `Prompt` 删掉后留下悬空绑定。
5. 不允许把“清理绑定”写成删除后的异步修复、后台补偿或人工处理步骤。
6. 后续修复文档时，必须把这条规则写入 `Prompt` 删除契约，同时与绑定契约、运行时契约、巡检契约保持一致。

### 4.5 已确认的 `Prompt` 文件路径与文件名规则

以下规则已经确认，后续线程直接执行，不再讨论：

1. `Prompt` 正文文件固定落在工作路径下的 `prompts/` 目录。
2. 第 7 章后续修订时，必须把这一点写成明确契约，不允许只写“创建 `.md` 文件”而不写落点目录。
3. `Prompt` 文件名规则直接服从系统设计文档中的文件命名规则，不再另发明一套局部规则。
4. 后续修订时，必须显式引用系统设计中已经确认的文件名允许字符、禁止字符、尾随限制、保留名限制。
5. 文件名非法时，返回 `422 + VALIDATION_ERROR`。
6. 路径唯一冲突时，返回 `409 + CONFLICT`。
7. 这里的“路径问题”不是目录位置有争议，而是接口契约必须把“固定落在 `prompts/`”和“文件名规则服从系统设计”写清。
8. 不允许后续线程把这条规则改写成模糊表述，例如“按实现决定文件名”“按运行环境处理文件名”。

### 4.6 已确认的 `workflow_node_actions` 更新配套规则

## 6. 明确禁止事项

后续修复线程不得做以下事情：

1. 不得保留 `link` 的占位兼容描述。
2. 不得保留“以后也许会支持，所以先留着”的模糊措辞。
