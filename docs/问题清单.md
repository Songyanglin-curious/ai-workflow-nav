# 个人AI工作流导航器-接口契约修订清单-v1

## 1. 文档定位

本文档用于给 [个人AI工作流导航器-接口与数据契约-v1](/D:/code/ai-workflow-nav/docs/个人AI工作流导航器-接口与数据契约-v1.md) 提供一版可直接执行的修订清单。

本轮目标：

- 只产出问题与修订建议清单
- 不改 SQL
- 不改实现代码
- 不直接改现有接口正文
- 让后续修订接口正文时，可以直接按本清单逐项落地

对照依据：

- [个人AI工作流导航器-数据库表结构设计-v1](/D:/code/ai-workflow-nav/docs/个人AI工作流导航器-数据库表结构设计-v1.md)
- [个人AI工作流导航器-系统设计-v1.1](/D:/code/ai-workflow-nav/docs/个人AI工作流导航器-系统设计-v1.1.md)
- [个人AI工作流导航器-模块设计-v1](/D:/code/ai-workflow-nav/docs/个人AI工作流导航器-模块设计-v1.md)
- `sql/schema/v1/tables/*.sql`

## 2. 本清单采用的默认决策

为避免清单本身继续遗留待决策项，本轮先采用以下默认修订方向：

1. `link` 不再作为 v1 的可写绑定类型。
2. `link` 仍可作为运行时展示语义保留，但不允许通过第 9 章写接口创建或更新。
3. `summaries` 在 v1 固定映射为 `insight_records.record_type = 'summary'`。
4. GET 接口不再把“读取即补齐目录入口/目录”定义为对外主语义。
5. `ProjectViewport` 的默认返回采用“保留默认视角返回能力，但新增 `source` 字段，并允许默认值场景下 `createdAt` / `updatedAt` 为 `null`”的方向，不通过 GET 懒创建记录。

## 3. 全局统一修订

### G01 统一工具配置术语

- 原问题：
  - 第 9 章把 `tool` 绑定目标写成配置中的 `toolKey`，但系统设计和其他文档统一使用 `tool_key` 表达配置真实键名，容易导致实现层把 DTO 风格命名和配置键名混为一体。
- 修订意图：
  - 把“配置项真实键名”和“HTTP DTO 字段命名风格”拆开。
  - 全文凡是表达配置真实引用时，统一写 `tool_key`。

## 5. 落地顺序建议

为减少来回返工，建议后续修订接口正文时按以下顺序落地：

## 6. 完成标准

- 每个 API 的请求、响应、错误码都能被现有表结构直接支撑，或明确声明为应用层行为
