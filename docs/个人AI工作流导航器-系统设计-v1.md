# 个人 AI 工作流导航器系统设计 v1

## 1. 文档说明

本文档用于整理当前版本的系统设计思路，目标是先把概念、边界、关系和风险梳理清楚，再进入数据表、接口和实现阶段。

当前状态定义如下：

- 已确定：可以作为后续设计与开发的基础约束。
- 建议采用：基于现有想法整理出的推荐方案。
- 待确认：当前草稿中仍有歧义或冲突，后续需要继续收敛。

---

## 2. 设计目标

本项目是一个围绕“个人 AI 工作流”进行组织、检索、可视化和落盘管理的工具，其核心目标包括：

1. 管理提示词、工作流、树结构、方案等多层对象。
2. 让树节点承载实际的工作上下文，并与工作流、对话记录、总结记录建立关联。
3. 通过本地服务统一管理业务数据，避免多入口直接操作文件导致混乱。
4. 让文件承担同步与恢复职责，让 SQLite 承担运行时元数据、关系和查询的数据源职责。
5. 兼容未来的多个壳层入口，例如浏览器插件、Trae 插件、独立页面。

---

## 3. 基础规则

### 3.1 文件命名规则

允许字符：

- 中文
- 英文
- 数字
- 下划线 `_`
- 短横线 `-`
- 空格
- 普通括号
- 多个点号 `.`

禁止字符：

- `<`
- `>`
- `:`
- `"`
- `/`
- `\`
- `|`
- `?`
- `*`

额外限制：

- 文件名不能以空格结尾
- 文件名不能以 `.` 结尾
- 禁用保留名：`CON`、`PRN`、`AUX`、`NUL`、`COM1~COM9`、`LPT1~LPT9`

### 3.2 唯一 ID 规则

- 所有领域对象都必须有自己的唯一 ID。
- 全局唯一 ID 统一使用 `UUIDv7` 生成。

### 3.3 数据库存储

- 使用 SQLite 存储索引、关系、元数据和查询加速信息。
- SQLite 是运行时元数据、关系和查询能力的唯一数据源。
- 文件类内容本身仍以磁盘中的实际文件为准。
- 数据库中涉及文件或文件夹的位置字段，统一保存相对路径，不保存绝对路径。

## 4. 设计原则

### 4.1 单一合法入口

所有业务数据的合法读写入口都应是本地服务，其他壳层只能通过接口访问。

### 4.2 分层调用

系统采用三层架构：

1. 上层交互层
2. 中层索引/服务层
3. 底层文件层

约束如下：

- 只能逐层调用
- 不能跨层直写

### 4.3 运行时与同步时的数据源分工

建议采用以下原则：

1. 运行时，SQLite 是元数据、关系和查询能力的唯一数据源。
2. 文件类内容本身以磁盘中的实际文件为准。
3. 数据同步只允许手动触发。
4. 同步阶段，导出的文件是同步过程中的唯一数据源。
5. 从 Git 拉取后的恢复过程，以文件作为恢复输入源。
6. 任何自动同步或自动覆盖行为默认禁止。

### 4.4 关系优先使用 ID

所有真实关联关系必须使用唯一 ID，不应使用名称进行关联。名称只用于显示、检索和人工理解。

## 5. 领域模型

### 5.1 提示词 Prompt

用途：管理单条提示词及其元数据。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `content_file_path`
- `created_at`
- `updated_at`

规则：

- 一份提示词对应一个 `.md` 文件。
- 名称必须符合文件命名规则。
- 正文保存在文件中，元数据与索引保存在 SQLite 中。

### 5.2 工作流 Workflow

用途：描述一个可复用的工作流对象。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `mermaid_source`
- `created_at`
- `updated_at`

规则：

- 工作流流程图使用 Mermaid 语法描述。
- Mermaid 中的节点是工作流节点的表示方式。

### 5.3 工作流节点动作绑定 WorkflowNodeActionBinding

用途：将工作流中的某个 Mermaid 节点绑定到一个可执行动作目标，而不只是绑定提示词。

建议字段：

- `workflow_id`
- `mermaid_node_id`
- `action_type`
- `target_ref`
- `created_at`
- `updated_at`

字段说明：

- `action_type`：动作类型，用于决定点击节点后应该执行什么行为。
- `target_ref`：动作目标引用，用于定位真正的业务对象或本地配置项。

建议支持的动作类型：

- `prompt`：复制或打开一个提示词
- `tool`：调用一个本地工具或外部工具
- `link`：打开一个链接或某类可跳转资源

建议规则：

- 节点允许不绑定任何动作。
- 当前每个 Mermaid 节点最多绑定一个主动作。
- 当 `action_type = prompt` 时，`target_ref` 指向 `Prompt.id`。
- 当 `action_type = tool` 时，`target_ref` 不直接存命令细节，而是指向本地配置中的工具定义键。
- 绑定依赖 `mermaid_node_id`，因此保存工作流时需要做绑定有效性校验。
- 真正的工具调用参数、路径、命令模板等细节应保存在本地配置中，而不是直接写入业务绑定表。

### 5.4 树 Tree

用途：承载一组树节点及其关系，用于知识组织和工作导航。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `created_at`
- `updated_at`

### 5.5 树节点 TreeNode

用途：承载具体工作上下文，是系统的核心使用单位。

建议字段：

- `id`
- `name`
- `description`
- `status`
- `created_at`
- `updated_at`

树节点可承载的业务内容包括：

- 名称与描述展示
- 关联一个工作流
- 展示节点关联工作流
- 触发工作流节点绑定动作
- 当节点动作为 `prompt` 时快速复制对应提示词
- 当节点动作为 `tool` 时调起对应本地工具
- 保存和查看对话记录
- 展示与该节点相关的总结/认知文件

### 5.6 树关系 TreeEdge

用途：表达树中的父子关系。

建议字段：

- `tree_id`
- `parent_node_id`
- `child_node_id`
- `sort_order`

建议说明：

- 树节点与树关系解耦。
- 树关系应采用“一条边一行”的形式存储。
- 不建议直接在一条记录中维护 `child_ids` 数组。

### 5.7 树节点与工作流绑定 TreeNodeWorkflowBinding

用途：将树节点与工作流建立关联。

建议字段：

- `node_id`
- `workflow_id`
- `created_at`
- `updated_at`

规则：

- 当前限定为树节点与工作流一对一绑定。
- 如果未来出现一对多需求，优先考虑拆分节点而不是扩大节点复杂度。

### 5.8 方案 Plan

用途：作为树的更上层组织抽象。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `created_at`
- `updated_at`

### 5.9 方案与树绑定 PlanTreeBinding

用途：表示方案与树之间的集合关系。

建议字段：

- `plan_id`
- `tree_id`
- `sort_order`

规则：

- 方案是树的集合管理层，不是树内容本身。
- 一棵树可以属于多个方案。
- 方案与树的真实关联必须通过 `tree_id` 维护，而不是通过树名称。

### 5.10 对话记录 ConversationRecord

用途：挂载到树节点上的 AI 对话记录文件夹入口，而不是单个文件入口。

建议字段：

- `id`
- `node_id`
- `folder_path`
- `created_at`
- `updated_at`

规则：

- `folder_path` 保存对话记录目录的相对路径。
- 对话记录以多个 `.md` 文件形式保存在该目录下。
- 支持一个节点下通过同一个目录承载多个对话文件。
- 文件名建议使用“时间戳 + 节点名称”。
- 默认向最新的对话记录文件追加内容，避免覆盖。
- 需要支持手动创建新的对话记录文件，避免单文件过大导致不可用。
- 前端需要提供“追加到最新对话文件”和“新建对话记录文件”的操作入口。
- 列表展示时，前端读取该目录下的文件列表。

### 5.11 认知/总结记录 InsightRecord

用途：记录用户对某个节点的理解、总结与沉淀，入口应为文件夹而不是单文件。

建议字段：

- `id`
- `node_id`
- `folder_path`
- `record_type`
- `created_at`
- `updated_at`

说明：

- `folder_path` 保存认知/总结目录的相对路径。
- 该类记录主要来自规则映射到的文件夹内容。
- 它们是节点长期认知沉淀的重要组成部分。

### 5.12 配置 Config

用途：存储系统运行相关配置。

建议内容：

- 数据存储根路径
- 本地服务端口
- 外部工具打开配置
- 本地工具定义注册表
- 未来可能需要的大模型 Key
- 其他运行参数

说明：

- 本地工具定义注册表用于维护工具键、启动方式、参数模板、目标路径规则等信息。
- 工作流节点动作绑定中的 `tool` 类型只保存对本地工具定义的引用，不直接保存命令细节。

## 6. 功能设计

### 6.1 提示词管理

支持：

- 新增
- 删除
- 修改
- 查询
- 快速复制正文

### 6.2 工作流管理

支持：

- 新增
- 删除
- 修改
- 查询
- Mermaid 编辑与渲染
- 节点与动作绑定维护

### 6.3 树与节点管理

支持：

- 树的增删改查
- 树节点的增删改查
- 树关系维护
- 树节点与工作流绑定
- 节点状态维护

### 6.4 节点相关能力

节点需要支持：

- 展示名称与描述
- 展示工作流名称
- 展示节点状态
- 展示当前位于工作流的哪个节点
- 触发工作流节点绑定动作
- 当节点动作类型为 `prompt` 时，支持快速复制对应提示词
- 当节点动作类型为 `tool` 时，支持调起对应本地工具
- 展示历史对话目录下的文件列表
- 展示总结/认知目录下的文件列表
- 调用外部工具打开关联文件

### 6.5 对话记录保存

当前建议优先支持以下方式：

1. 将剪贴板内容追加进最新的对话记录文件。
2. 在节点上手动新建对话记录文件。

前端接口语义建议：

- 读取节点对话目录信息
- 读取对话目录下的文件列表
- 追加内容到最新对话文件
- 在对话目录下创建新的对话记录文件

以下能力建议后续阶段再做：

- 浏览器页面中的 AI 对话结构化提取
- 多网站适配
- AI/人角色识别
- 多选追加写入

原因：

- 不同网站的 DOM 结构差异很大。
- 插件适配维护成本高。
- 该功能更适合作为第二阶段或适配器体系来做。

### 6.6 孤儿数据与异常巡检

系统需要支持巡检以下问题：

- 有索引或路径但文件不存在
- 文件存在但索引缺失
- 工作流节点未绑定动作
- 树节点未绑定工作流
- 绑定关系引用了不存在的对象
- `tool` 类型动作引用了不存在的本地工具定义

## 7. 节点状态设计

本项目正式采用“持久状态”和“运行时状态”分离的设计。

“当前节点”不属于节点本身的长期业务属性，因此不能和“待处理/待修正/默认”放在同一个状态字段里。

### 7.0 设计结论

节点状态分为两类：

1. 持久状态：表达节点本身的长期业务状态。
2. 运行时状态：表达当前界面或当前操作上下文中的聚焦节点。

### 7.1 持久状态

用于表达节点本身的长期业务状态，存储在节点相关的数据中。

采用以下状态值：

- `default`：无标记
- `todo`：要处理节点
- `fix`：要修正节点

规则：

- 一个节点同一时刻只能有一个持久状态。
- 持久状态由人手动判断和切换。
- 持久状态用于降低记忆成本，帮助快速识别待办和异常节点。

### 7.2 运行时状态

用于表达用户当前正在操作或聚焦的节点。

- 采用以下规则：

- 以 `active_node_id` 形式单独维护
- 以 `active_workflow_node_id` 形式维护当前位于工作流的哪个节点
- 它不属于节点主状态字段
- 允许为空，表示当前没有选中节点
- 它应与具体视图或当前树上下文关联，而不是直接写入节点主表

实现建议：

- 如果当前上下文是单棵树，可按 `tree_id -> active_node_id` 维护。
- 如果节点已关联工作流，可在当前节点上下文中维护 `active_workflow_node_id`。
- 如果后续存在多个画布或多个视图实例，再单独扩展视图级运行时状态模型。

### 7.3 设计收益

- 业务状态和界面状态分离后，语义更稳定。
- 节点不会因为用户临时选中而发生业务数据变化。
- 前端切换当前节点时，不需要改动节点本身的长期状态字段。

## 8. 页面与交互设计

当前浏览器插件需要支持三类页面或视图。

### 8.1 维护页面

用于管理以下对象：

- 提示词
- 工作流
- 树
- 节点
- 方案
- 配置

### 8.2 树关系可视化页面

使用 React Flow 进行展示。

设计要求：

- 默认按方案进行展示
- 需要兼容没有方案的树
- 对于无方案树，可以在前端构造“虚拟方案”以统一语义和展示结构

当前版本先采用以下模式：

1. 一个画布仅渲染一棵树。
2. 通过左侧树列表切换当前展示的树。

说明：

- 该模式更容易控制复杂度，也更适合作为第一版实现。
- 多树同画布的方案暂不作为当前版本目标，后续如有明确需求再扩展。

### 8.3 节点展示模式

节点建议支持两种展示模式：

1. 默认模式
2. 详细模式

默认模式建议展示：

- 名称
- 描述
- 状态图标
- 历史对话目录下的文件列表
- 总结目录下的文件列表
- 工作流名称
- 当前节点主动作摘要

详细模式建议增加展示：

- 工作流节点详情
- 工作流渲染结果
- 工作流相关文件列表
- 总结文件列表的更多上下文
- 可点击的节点动作入口
- 节点动作的类型与目标信息

### 8.4 悬浮窗

悬浮窗用于承载节点的快速操作与上下文查看。

建议支持：

- 名称与描述展示
- 工作流渲染结果展示
- 点击触发节点动作
- 当节点动作为 `prompt` 时复制提示词
- 当节点动作为 `tool` 时调起本地工具
- 历史对话目录下的文件列表
- 总结目录下的文件列表
- 调用外部工具打开文件
- 快速切换到带状态标记的节点
- 快速定位到树中的对应节点
- 可拖拽
- 可折叠

## 9. 存储设计

### 9.1 存储分层

底层文件分为两类：

1. 配置型文件
2. 工作流内容文件

配置型文件包括：

- 数据存储根路径
- 模型 Key
- 外部工具配置
- 本地服务端口

工作流内容文件包括：

- 提示词文件
- 树相关文件
- 节点相关文件
- 对话记录
- 个人总结

### 9.1.1 当前工作目录结构

当前版本建议采用如下工作目录结构：

```text
工作目录/
  db/
    runtime.sqlite
    template.sqlite
  dbSyncs/
    manifest.json
    prompts.csv
    workflows.csv
    workflow_node_actions.csv
    trees.csv
    tree_nodes.csv
    tree_edges.csv
    tree_node_workflows.csv
    plans.csv
    plan_trees.csv
    conversations.csv
    insights.csv
  prompts/
    prompt-name__shortId.md
  trees/
    tree-name__shortId/
      node-name__shortId/
        chatLogs/
        summaries/
  local.config.jsonc
```

说明：

- `db/` 是数据库文件夹，而不是单个数据库文件名。
- `runtime.sqlite` 是运行时数据库文件。
- `template.sqlite` 是数据库模板文件。
- `dbSyncs/` 用于存放数据库导出的同步文本文件。
- `prompts/` 用于存放提示词正文文件。
- `trees/` 用于存放树相关的真实落盘内容。
- `local.config.jsonc` 用于存放本地配置。

### 9.1.2 树与节点目录命名规则

树实例文件夹和节点实例文件夹采用：

- `初始名称 + "__" + shortId`

例如：

- `nginx-usage__01JXYZ`
- `install-node__01JABC`

规则说明：

- 文件夹名称在对象创建时生成。
- 生成后将该目录名写入数据库。
- 后续即使对象名称修改，目录名也不跟着变化。
- 真实关联仍然全部通过 ID 维护，不通过目录名维护。

这样设计的好处是：

- 目录具备可读性
- 可避免重名冲突
- 可降低长名称和特殊字符带来的风险
- 目录路径一旦生成就可以长期稳定使用

### 9.1.3 节点子目录命名

节点目录下当前约定的固定子目录为：

- `chatLogs`
- `summaries`

说明：

- 原先的 `summarys` 已确认改为 `summaries`。
- 这两个目录名称属于固定名称，不跟随节点名称变化。

### 9.1.4 当前为什么没有 `workflows/` 目录

当前版本不单独建立 `workflows/` 目录。

原因：

- 工作流的元数据、关系和 `mermaid_source` 当前都存放在运行时数据库中。
- 工作流同步时通过 `dbSyncs/workflows.csv` 等表级导出文件参与同步。
- 当前工作流本身没有独立的正文型落盘目录需求。

补充说明：

- 如果后续工作流开始承载附件、模板文件、截图、说明文档等真实落盘内容，再新增 `workflows/` 目录即可。

### 9.2 文件映射规则

当前版本已确认采用“目录约定为主，数据库记录相对路径，`dbSyncs` 承担同步导出”的映射基线。

当前确认的规则如下：

1. 数据库中的文件路径和文件夹路径统一存为相对路径。
2. 真实文件内容按目录约定落盘，不通过数据库字段直接拼接磁盘结构。
3. 树、节点、提示词等真实文件内容的落点必须稳定且可预测。
4. 树和节点的目录名在创建时生成，并写入数据库，后续不随名称修改而变化。
5. 节点固定子目录当前采用 `chatLogs/` 和 `summaries/`。
6. 提示词正文文件落在 `prompts/` 目录下。
7. 工作流当前不单独建立内容目录，运行时内容由数据库承载，同步时通过 `dbSyncs/` 中的表级文件导出。
8. `dbSyncs/` 采用“一个表一个文件”的多文件导出方案，并保留 `manifest.json` 作为全局说明文件。

扫描恢复原则：

- 运行时以 SQLite 为元数据、关系和查询能力的唯一数据源。
- 当需要从磁盘恢复时，可基于目录约定和同步导出文件重建索引。
- 目录约定负责定位真实文件内容。
- `dbSyncs/` 负责恢复结构化元数据与关系。

当前阶段已经足够支撑后续表结构、接口和实现设计。

## 10. 同步设计

### 10.1 当前建议结论

建议明确以下同步原则：

1. 运行时，SQLite 是元数据、关系和查询能力的唯一数据源。
2. 文件类内容本身以磁盘中的实际文件为准。
3. 同步只允许手动触发。
4. 在同步阶段，导出的文件是同步过程中的唯一数据源。
5. 从 Git 拉取后的恢复过程，以文件作为恢复输入源。
6. 默认禁止任何自动同步、自动覆盖或自动回写行为。

### 10.2 导出文本文件的目的

这里需要更准确地区分“运行时”和“同步时”两种状态。

运行时规则：

- SQLite 是元数据、关系、索引等信息的唯一运行时数据源。
- 文件类内容本身仍以磁盘中的实际文件为准。

导出/同步时规则：

- 在落盘和向 GitHub 推送时，由数据库生成用于同步的文本文件。
- 在这次同步动作中，导出的文件是唯一真实数据源。
- 在从 Git 拉取并进行同步恢复时，文件是唯一数据源。

简化后的结论：

- 运行起来之后，SQLite 是运行时唯一数据源。
- 进行同步时，文件是同步阶段的唯一数据源。
- 文件的核心职责是同步与恢复，而不是替代运行时数据库。

## 11. 架构设计

### 11.1 总体架构

项目整体架构为：

上层操作层  
↓  
中层索引/服务层  
↓  
底层文件层

### 11.2 上层交互层职责

上层只表达“我要做什么”，例如：

- 新建节点
- 关联工作流
- 保存对话
- 读取节点详情

上层不直接接触：

- 文件
- 目录结构
- 路径拼接

### 11.3 中层服务层职责

中层是唯一调度层，负责：

- 生成 ID
- 更新索引
- 维护关系
- 决定文件写入顺序
- 调用扫描器
- 返回统一数据结构

中层不负责：

- 页面展示
- 浏览器 DOM 提取
- 自由拼接磁盘正文格式

### 11.4 底层文件层职责

底层只负责：

- 读文件
- 写文件
- 列目录
- 改名
- 删除
- 判断文件是否存在

底层不理解：

- `node` 是什么
- `workflow` 是什么
- 关系是否合理

### 11.5 该架构的价值

- 只有一套合法业务读写入口
- 可以显著降低复杂度
- 后续切换前端壳层时，接口可以保持稳定
- 浏览器插件、Trae 插件、页面都只是交互入口

## 12. 当前已收敛的关键设计点

### 12.1 工作流存表 vs 文件系统为真实数据源

这里的结论是：

- 元数据与关系在运行时由数据库作为唯一数据源。
- 文件类内容始终以磁盘中的实际文件为准。
- 在同步阶段，导出的文件承担唯一同步数据源的角色。

说明：

- 之前的矛盾主要来自表述不准确，而不是设计目标真的冲突。
- 工作流本身的结构、元数据和关系在运行时可由数据库承载。
- 与工作流相关的文件类资源仍然遵循“磁盘文件是真实内容源”的原则。

### 12.2 方案通过树名称关联 vs 关系统一用 ID

该问题已确认收敛为：

- 所有对象都必须有唯一 ID。
- 所有真实关联都统一使用 ID。
- 名称仅用于显示、检索和人工理解，不参与真实关联。

### 12.3 当前节点状态 vs 业务状态

该问题已确认收敛为：

- 节点业务状态和当前节点运行时状态拆开管理。
- 具体规则以第 7 节为准。

### 12.4 树关系字段混杂

该问题已确认收敛为：

- 树关系只表达结构关系。
- 当前选中节点单独维护。
- 不在树关系结构中混入运行时字段。

### 12.5 Mermaid 节点标识的稳定性

工作流节点动作绑定依赖 `mermaid_node_id`，因此维护顺序需要明确。

采用以下规则：

1. 先维护工作流本身。
2. 再维护工作流节点与提示词或其他动作单元的关系。
3. 保存节点绑定关系时，需要基于当前 Mermaid 节点集合做差异比对。
4. 对于已经失效的旧绑定关系，需要清理为垃圾关系，避免残留。

说明：

- 当前版本先采用“基于 diff 清理失效关系”的方案。
- 是否为工作流节点引入更稳定的中间标识，可留到后续版本再评估。

## 13. 当前已确认决策与剩余注意事项

1. 删除采用级联删除；重命名不影响关系，因为所有关联都使用 ID。
2. 树节点不允许跨树复用，避免结构和语义混乱。
3. 排序字段统一采用 `sort_order`。
4. 同步策略已确认采用人工控制、允许清空重建、不做自动回滚。
5. 文件映射规则已有当前版本可落地的目录基线。
6. 浏览器插件抓取多网站 AI 对话暂不在当前版本实现，后续再处理。
7. 外部工具打开文件的配置协议已定稿。
8. 节点状态已经与当前节点运行时状态拆开，其他 UI 状态继续保持不进入业务主表。

### 13.1 排序字段建议

该方案已确认采用：

- 所有需要排序的同级对象都使用 `sort_order INTEGER`。
- 排序语义统一为“数值越小越靠前”。
- `sort_order` 只在同一父级范围内有意义，不做全局排序。
- 为了方便后续插入，可默认按较大步长分配，例如 `1000`、`2000`、`3000`。

建议适用范围：

- 同一父节点下的子节点顺序
- 同一方案下的树顺序
- 同一列表中的手动排序对象

### 13.2 同步冲突的含义

这里的“冲突”主要不是 Git 冲突本身，而是数据源切换时的状态不一致，例如：

- 运行中的 SQLite 还是旧数据，但磁盘上的同步文件已经被 Git 拉取更新
- 数据库刚导出同步文件后，用户又继续修改了运行时数据
- 导入同步文件时，数据库中已有一部分旧索引或旧关系尚未清理

因此后续需要明确：

已确认规则：

- 由人决定何时导入。
- 由人决定何时导出。
- 导入时允许先清空再重建。
- 导入失败时不做自动回滚，由人排错后重新导入。
- 页面提供刷新按钮，用于主动获取最新数据。
- 导入完成后由人手动刷新页面即可。

说明：

- 当前使用频率较低，这套人工控制流程是可接受的。
- 该策略优先保证实现简单、行为可控，而不是追求自动化回滚。

### 13.3 外部工具配置协议建议

该方案已确认定稿。

配置文件格式：

- 使用 `JSONC` 单文件配置。
- 推荐文件名为 `external-tools.jsonc`。

配置结构：

- `version`
- `defaults`
- `tools`
- `routes`
- `security`

设计原则：

- 业务表中只保存对 `tool_key` 的引用。
- 本地配置中保存工具的执行细节。
- 工具定义与默认路由分离。
- 执行时使用 `exePath + args[]`，不存整条命令字符串。

`tools` 负责定义可执行工具，建议字段包含：

- `tool_key`：工具唯一键
- `label`：显示名称
- `enabled`：是否启用
- `platform`：适用平台，可选
- `exePath`：可执行文件绝对路径
- `actions`：动作定义集合

`actions` 建议支持：

- `openFile`
- `openFolder`
- `openPath`
- `openAtLine`

每个 action 建议包含：

- `args`：字符串数组
- `requiresExists`：是否要求目标存在

占位符建议固定为：

- `{filePath}`
- `{folderPath}`
- `{path}`
- `{line}`
- `{column}`

`routes` 负责默认路由策略，而不是替代业务绑定：

- 当业务层已经通过 `tool_key` 指定工具时，优先使用业务绑定。
- 当业务层没有显式指定工具时，再根据 `routes` 做默认匹配。
- 当前版本 `routes.match` 先只支持按扩展名匹配。

`security` 建议包含：

- `allowAbsolutePaths`
- `allowedRoots`

安全要求：

- `allowAbsolutePaths` 默认建议为 `false`。
- 业务数据层继续只传相对路径。
- 服务层在执行前将相对路径解析为绝对路径。
- 所有目标路径都必须经过规范化、解析和允许根目录校验。
- `exePath` 允许为本机绝对路径。

执行模型：

1. 校验目标路径是否合法。
2. 优先读取业务绑定的 `tool_key`。
3. 若无业务绑定，再按 `routes` 匹配默认工具。
4. 从工具定义中取出对应 action。
5. 替换参数模板中的占位符。
6. 使用 `spawn(exePath, args)` 执行。

说明：

- 这样可以避免业务数据污染本地环境细节，也方便不同机器做各自配置。
- `routes` 只负责默认兜底，不负责表达业务语义。

### 13.4 领域状态与 UI 状态的边界

这里指的是：

- 领域状态：会进入业务数据、需要长期保存、会参与同步的数据
- UI 状态：只服务于当前界面交互、不应该写入业务模型的数据

例如：

- 节点的 `todo/fix/default` 属于领域状态
- 当前选中的节点 `active_node_id` 属于运行时状态
- 当前工作流执行到的节点 `active_workflow_node_id` 属于运行时状态
- 当前展开了哪个面板、悬浮窗是否折叠、当前过滤关键词、当前缩放比例，这些都属于 UI 状态

要求：

- UI 状态不要混进领域模型主表
- 只有真正需要长期保存和同步的状态，才进入业务数据层

该划分已确认采用。

## 16. 当前结论

基于现有设计，当前最关键的三项结论是：

1. 运行时由 SQLite 承担元数据、关系和查询能力的唯一数据源职责，同步阶段由导出文件承担唯一同步数据源职责。
2. 所有真实关系必须统一使用唯一 ID。
3. 节点的业务状态与运行时状态必须拆开管理。

在这三项原则固定后，后续再展开数据库表结构、接口设计和页面交互，会更稳定，也更不容易反复推翻。
