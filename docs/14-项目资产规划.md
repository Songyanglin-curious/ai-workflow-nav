# 项目资产规划

## 目标

`assets/` 用来沉淀项目级长期资产。

这类内容不直接服务某一次功能实现，而是会在后续多个任务中重复复用，用来减少重复沟通、统一判断边界、稳定协作方式。

## 边界

- `docs/`：当前项目这次要做什么、为什么这样做、实现需要改哪些层
- `assets/`：以后类似任务还会反复复用的规则、方法、模板、排障手册、检查清单
- `web/` / `server/` 内局部文档：只服务该端某段代码实现的局部说明

## 最小化目录架构

当前阶段先只保留按资产类型划分的一层目录，不提前按 `web/`、`server/` 分层。

```text
assets/
├─ rules/        # 长期规则与稳定偏好
├─ skills/       # 可重复执行的方法与工作流
├─ playbooks/    # 排障与操作手册
├─ templates/    # 可复用模板
└─ checklists/   # 审查、设计、交付检查清单
```

这样处理的原因：

- 当前沉淀量很少，先分端会提前制造目录复杂度
- 很多资产天然跨前后端，例如文档优先、范围收敛、Windows 原生依赖排障
- 第一层按类型分，后续更容易累计，也更符合 `assets/README.md` 现有约定

## 分层规则

### 第一层固定按资产类型分

- 规则类放 `assets/rules/`
- 工作流类放 `assets/skills/`
- 排障手册放 `assets/playbooks/`
- 模板放 `assets/templates/`
- 检查项放 `assets/checklists/`

### 第二层暂不默认分端

只有满足下面条件之一，才继续在子目录中区分 `web/`、`server/` 或 `common/`：

- 同类型资产累计到 5 个以上，且已经出现明显的前后端聚类
- 该资产明显强绑定某一端，跨端复用价值很低
- 该资产需要贴着某端长期维护，继续混放会降低可读性

如果暂时不满足这些条件，优先继续平铺。

## 累计规则

### 进入 `assets/` 的条件

只有满足下面条件的内容，才进入 `assets/`：

- 已经在至少两个任务中复用，或者明显会在后续反复复用
- 已经能形成稳定边界，不再依赖某个单一业务页面
- 内容表达的是规则、方法、模板、检查项、排障步骤，而不是某次业务实现细节

### 不进入 `assets/` 的内容

- 当前模块这次要怎么做，继续留在 `docs/`
- 某个页面、某个接口、某张表的局部设计，不提前抽成资产
- 还在快速变化、没有稳定方法论的业务细节

### 与 `AGENTS.md` 的关系

- `AGENTS.md` 只放仓库级硬规则和高频协作约束
- `assets/` 放可以长期累计、可继续展开的资产正文
- 某条资产被长期验证稳定后，再考虑回收进对应的 `AGENTS.md`

## 当前首批累计方案

结合当前已经明确的两组沉淀内容，先累计下面两类资产：

### 规则资产

文件：`assets/rules/前端页面实现偏好.md`

用途：

- 沉淀已经多次出现、但还不适合直接塞满 `web/AGENTS.md` 的前端页面实现偏好
- 先作为项目级长期规则资产累计，后续稳定后再决定是否提升为端级硬规则

当前纳入的内容：

- Naive UI 优先，少造基础组件
- 页面高信息密度、紧凑布局、对齐优先
- 图标按钮优先配合 tooltip
- 危险操作统一红色与中文二次确认
- 页面清理时主动删除未使用组件、样式、theme、provider
- 容器内滚动优先，工作台布局优先固定头部加内容区滚动

### 工作流资产

文件：`assets/skills/项目级工作流清单.md`

用途：

- 把当前已经识别出的稳定方法沉淀为后续可继续拆分的 skill 候选
- 先收敛成一个清单文件，避免在资产很少时过早拆出大量细文件

当前纳入的内容：

- `docs-driven-change`
- `scope-pruning`
- `server-module-bootstrap`
- `native-dependency-troubleshoot-windows`
- `error-contract-unifier`
- `dictionary-module-design`
- `vue-naive-page-builder`
- `vue-page-cleanup`
- `workspace-two-pane-layout`

## 本轮落地要求

本轮只做“小而完整”的一步：

1. 保持 `assets/` 一级目录简单
2. 为每类目录建立最小 README 或占位说明
3. 把已经稳定的两组沉淀整理进规则资产和工作流资产
4. 不提前把所有工作流拆成独立 skill 文件

后续只有在单类资产继续增长、边界已经稳定时，再做二次拆分。
