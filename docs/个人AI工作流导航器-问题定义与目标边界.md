
# 个人AI工作流导航器-Personal AI Workflow Navigator
## 规则名词表
###  文件命名规则
可用：中文、英文、数字、下划线 _、短横线 -、空格、普通括号、多个点号
禁用：< > : " / \ | ? *
禁尾：末尾空格、末尾 .
禁名：CON PRN AUX NUL COM1~COM9 LPT1~LPT9
## 唯一ID
全局的唯一ID 使用UUIDv7 进行生成
## 数据库
SQLite数据库进行存储
## 功能描述

1. 提示词的管理功能 （增删改查）
    1. 一份提示词使用一个.md文件进行存储
    2. 提示词的元数据使用 一张表进行存储
    3. 元数据包含 唯一ID 名称 描述 标签 类别
    4. 名称 要符合文件命名规则
2. 工作流的自定义功能 （增删改查）
3. 工作流使用mermaid这样的工具编辑流程图
     1. 工作流的存储方式使用一张表进行存储
     2. 一个工作流就是一个对象 包含 工作流的元数据 与 工作流的流程图
     3. 工作流需要的字段是 唯一ID 名称 描述 标签 类别 流程图
     4. 所谓的流程图就是mermaid的语法
     5. 工作流的节点就是mermaid的语法中的节点
4. 工作流的节点与提示词进行关联
    1. 工作流的节点 要支持 与 提示词 进行关联 目前限定为 一对一关联 其次可以允许节点不关联提示词
    2. 关联关系的存储方式使用一张表进行存储
    3. 关联关系的字段是 工作流唯一ID 提示词唯一ID  工作流节点标记 一般而言就是mermaid语法中的节点标记
5. 能够从提示词和工作流节点快速复制提示词内容到剪贴板
    1. 点击节点 可以复制节点的提示词内容到剪贴板
6. 需要一个树关系的图结构展示工具进行可视化展示 使用Vue Flow进行可视化展示
    1. 树的每个节点都是由人来决定增删改查
    2. 每个树的节点的拥有一个唯一ID
    3. 节点承担的功能
        - 唯一ID
        - 名称显示 
        - 描述显示 （名称和描述从宏观表明了节点的功能和作用）
        - 节点可以关联工作流（目前只支持一对一关联 ，如果要一对多关联那么就应该拆节点，避免膨胀）
        - 节点的唯一ID 与 工作流唯一ID进行关联
        - 关联关系使用一张表进行存储
        - 节点关联的工作流功能 主要是显示工作流 与快速复制工作流节点对应的提示词
        - 节点上的工作流 需要再加一个显示状态 就是突出下当前在哪个节点 这个完全由人来判断与操作 （人可以点击节点来切换当前节点，也要支持都不选中的状态与操作）
        - 对话保存的记录入口 （使用AI的对话保存为md文件,从这里面能够快速调用工具打开对话文件）
        - 对话的保存我允许存多个文件，使用时间戳加节点名称命名
        - 保存对话的操作如下默认是像最新的一个md文件追加写入（避免覆盖）,node支持创建新的对话记录md避免一个文件几百MB完全用不了。
        - 对话记录的查看我最后希望是配置外部的工具去打开查看，我自己不想写
        - 保存对话记录的操作有以下几个方式
            1. 将我粘贴板的内容追加进 最新的对话记录文件
            2. node添加保存对话的按钮，点击后各个网站的AI对话记录添加多选框 选择后将对应的对话记录追加进最新的对话记录文件（这个要支持多选与人员识别就是AI还是人 目前就两个角色）
        - 节点还需要一个展示我对这个节点认识的记录也是读取规则映射出来的文件夹内的文件，这是我真正的认识记录
        - 节点需要有个状态的标记 状态分为4中 当前的节点 要处理节点（正常） 要修正的节点（异常） 默认状态（无标记） 这个是为了降低对节点的记忆成本  也都是由人来决定的 
7. 树的节点关系 与 节点要解耦 仅仅通过节点ID 进行关联 节点的名称 与 关联关系的存储方式使用表进行存储 节点承载具体的内容 关联关系只需要 维护当前节点的ID 父ID 子IDs
8. 树需要有管理功能 （读取、创建、更新，删除需走保护动作） 树也需要有标签 名称 描述 等元数据 也是用表储存 
9. 还需要一个更上层的管理抽象 方案 方案 有自己的 名称 描述 标签 类别 等元数据 用表储存 
10. 方案 与树的关系 方案就是对树的名称的 进行集合管理 树才是真正的实体 方案仅仅只是关系 所以一棵树可以同时存在两个方案内 比如NGINX的使用树 可以存在很多的方案内的 因为他们仅仅是通过名称 进行关联的 所以完全不影响
11. 方案的管理功能 （增删改查）
12. 底层的文件我目前分为两类 
    - 配置型的文件  比如 数据储存的根路径  还有未来可能添加的大模型key等信息  还有各个外部工具打开需要的配置 比如 vscode  trae  还有本地服务的端口等
    - 工作流文件 这就是 提示词 工作流 树 节点 对话记录 个人总结等文件 这个我后面会建立一套映射规则 进行文件的存储  按照这个规则读取磁盘就能提供对应的接口数据。 文件夹的相对路径我会存表里面。
13. 一些辅助性的设计 
    1. 要检索出游离于现有关系的 文件 与 各种层级的名称
        -  有索引或者路径但是没有对应的文件夹或者文件的要展示出来
        - 工作流节点 没有对应的提示词或者工作流 要展示出来

14. 展示的设计 目前我们这是一个浏览器插件 我需要三种页面  
    1. 是各种维护页面比如 提示词 工作流 等
    2. 树关系的可视化页面 这个默认其实是 方案的展示界面但是有的时候树它没有方案 所以要能兼容 没有方案的情况 方案的展示分为两种情况 多棵树在一个画布 通过 左侧的树名称进行定位
    另一种是仅支持一个画布一棵树 通过左侧切换
    对于没有方案的树 我们可以在前端 做一个虚方案的抽象这样就可以保证语义和格式的统一 所以还是一套方案的展示界面
    3. 对于node节点有两种展示格式
        - 一种是默认的 名称 描述 状态图标 两个 文件列表 工作流名称  这个里面都只展示最基础的
        - 一种是 详细的 名称 描述 状态图标 两个 文件列表 工作流名称  这个里面展示的内容要更多一些 比如 工作流的节点  工作流的文件列表  总结的文件列表  还有工作流渲染的要清晰 好看 可点击 等
    3. 是一个悬浮窗 这个悬浮窗主要是展示node相关的信息 比如 名称 描述 工作流（渲染后的效果，可点击粘贴提示词） 历史对话的文件列表 总结的文件列表 点击后 调用对应的外部工具打开文件  这个悬浮窗要包含一个node所有本身的功能与操作 还有要附带一写其他的功能 比如 快速切换到标记了状态的的node，快速打开并定位到树中的状态，当然已经打开了包含这个树的页面就直接定位  等 未来也许还有其他的功能。
    4. 悬浮窗可拖拽 可以收起来

16. 项目整体框架 按照唯一后端接口服务 加插件前端 的架构
    上层操作层
    ↓
    中层索引/服务层
    ↓
    底层文件层

    只能逐层调用
    不能跨层直写

    1. 上层只表达“我要做什么”

    例如：

    新建节点
    关联工作流
    保存对话
    读取节点详情

    它不直接碰：

    文件
    目录结构
    路径拼接

    2. 中层负责规则与一致性

    它是唯一调度层，负责：

    生成 id
    更新索引
    维护关系
    决定文件写入顺序
    调用扫描器
    返回统一数据结构

    它不负责：

    页面展示
    浏览器 DOM 提取
    具体磁盘文件内容格式的自由拼接

    3. 底层只负责文件读写

    它只做：

    读文件
    写文件
    列目录
    改名
    删除
    判断文件是否存在

    它不理解：
    node 是什么
    workflow 是什么
    关系是否合理

> 这么设计的好处是只有一份状态 只有一份操作文件的接口 可以极大的降低复杂度；并且后续做trae的插件 只需要做trae的插件的壳就行 接口完全不用变。
所有壳层（浏览器插件、Trae 插件、页面）都只是交互入口；
业务数据的唯一合法读写入口是本地服务；
本地工作区文件是最终事实源。
17. 数据库文件化同步
 由于我们使用sqlite 存我们的关系 与 元数据 等信息 ；他是二进制数据 而 git对于二进制文件的处理不是很友好 。
 加一个数据导出 的  导出成文本文件 git 将这个文件进行管理  再来一个 文件导入数据库的
 我手动操作什么时候数据库映射文件什么时候文件映射到数据库，这个最多加一个页面操作的按钮 具体的时机完全由我来决定。
 也就是说这份数据库的文件 才是真正的最终的数据；数据库里面的是一份 运行时索引 / 加速层 可以类似于内存； 落盘的才是最终的产物。

 1. 文件系统是真实数据源
2. 所有正文内容以文件为准
3. SQLite 仅作为本地索引和查询层
4. 数据同步只允许手动触发
5. 默认方向是：文件 → SQLite
6. 任何自动写回文件的行为默认禁止
