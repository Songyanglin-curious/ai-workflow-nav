## 核心设计原则

### 单一合法入口

所有业务数据的合法读写入口都应是本地服务。
当前版本的网站前端只是交互壳层，不能绕过服务层直接操作数据库或磁盘。

### 分层调用

系统架构为三层

1. 上层交互层 (网页、插件等)
2. 中层服务层 (处理业务逻辑、调用本地服务)
3. 底层文件层 (存储提示词正文、配置文件、sqlite存元信息 等)

约束如下：

* 只能逐层调用
* 不能跨层直写

### ID 优先

所有领域对象都必须有唯一 ID，所有真实关联都统一使用 ID。
名称仅用于显示、检索和人工理解，不参与真实关联。

### 运行时与同步时分离

运行时和同步时的数据源职责不同：

- 运行时，SQLite 是元数据、关系和查询能力的唯一数据源。
- 文件类内容本身始终以磁盘中的实际文件为准。
- 同步阶段，导出的文本文件是同步过程中的唯一数据源。
- 从 Git 拉取后的恢复过程，以同步文件和磁盘文件作为恢复输入源。

### 手动同步

同步只允许手动触发，不做自动同步、自动覆盖或自动回写。
导入导出何时执行，由人决定。

## 基础规则

### 文件命名规则

文件名与标题一致。

允许字符：

- 中文
- 英文
- 数字
- 下划线 `_`
- 短横线 `-`
- 空格
- 普通括号
- 点号 `.`

禁止字符：

- `<`
- `>`
- `:`
- `"`
- `/`
- `\`
- `|`
- `?`
- `*`

额外限制：

- 文件名不能以空格结尾
- 文件名不能以 `.` 结尾
- 禁用保留名：`CON`、`PRN`、`AUX`、`NUL`、`COM1~COM9`、`LPT1~LPT9`
- 后续如果修改名称，应同步修改文件名
- 后续如有需要，可单独补名称校验能力

### 唯一 ID 规则

- 所有领域对象都必须有唯一 ID
- 全局唯一 ID 统一使用 `UUIDv7`

### 路径规则

- 数据库中涉及文件路径和文件夹路径的字段，统一存相对路径
- 不在业务数据层存绝对路径
- 绝对路径只在本地配置和实际执行阶段出现
