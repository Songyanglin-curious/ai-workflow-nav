# 个人 AI 工作流导航器技术路线与技术选型 v1

## 1. 文档定位

本文档用于承接当前已确认的系统设计与模块设计，给出 v1 阶段的技术路线与技术选型结论。

本文档主要回答以下问题：

- 当前版本系统采用什么运行形态
- 前端、服务层、数据库、文件存储分别采用什么技术
- 为什么这些技术与当前设计最匹配
- 这些选型带来的约束与代价是什么
- 当前不采用哪些常见方案，以及原因是什么

本文档不解决：

- 具体 API 字段命名
- 具体 SQL 建表脚本
- 单个页面的组件结构
- 单个任务如何拆分为 AI 执行卡

---

## 2. 设计依据

本技术选型基于以下已确认设计约束：

- 当前版本仅实现网站前端页面，不实现插件形态下的专属交互
- 所有业务数据的唯一合法读写入口是本地服务
- SQLite 是运行时元数据、关系与查询能力的唯一数据源
- 正文内容、推敲记录、总结记录等真实内容以磁盘文件为准
- 同步阶段以 `dbSyncs/` 下的导出文件作为唯一同步输入输出
- 同步采用手动触发，不做自动同步、自动覆盖或自动回写
- 工作流采用 Mermaid 文本描述
- Project / Node 页面中的图形区域定位为“树关系导航视图”，而不是复杂图编辑器
- 前端本地运行时状态与共享结构化数据分离

---

## 3. 技术路线结论

当前版本采用以下总体技术路线：

1. 网站前端作为唯一交互壳层
2. 本地 HTTP 服务作为统一业务入口
3. SQLite 承载运行时元数据、关系与查询
4. 文件系统承载正文与真实内容
5. CSV + `manifest.json` 承载手动同步数据
6. TypeScript 作为前后端统一主语言

这意味着当前版本是一个“本地服务 + 网站前端”的单机系统，而不是纯前端应用，也不是插件优先系统。

---

## 4. 技术选型结论

### 4.1 主语言与运行时

- 主语言：`TypeScript`
- 服务运行时：`Node.js LTS`
- 包管理：`pnpm`

选择原因：

- 当前仓库已经具备 TypeScript + Vite 前端原型基础，继续使用 TypeScript 成本最低
- 前后端统一语言有利于共享 DTO、校验规则和错误语义
- Node.js 更适合承接本地服务、文件访问、工具调用与 SQLite 集成

### 4.2 前端技术栈

- 前端框架：`Vue 3`
- 构建工具：`Vite`
- 路由：`Vue Router`
- 状态管理：`Pinia`
- 树与节点可视化：`@vue-flow/core`
- 工作流渲染：`mermaid`
- ui组件库：`Naive UI`



选择原因：

- 当前项目的主要复杂度在节点内部业务信息、维护页面与详情面板，而不是复杂图编辑器能力
- 图形区域当前定位为“导航视图”，核心需求是承载树关系、定位节点和维护父子关系
- Vue 3 更符合当前主要维护者的熟悉栈，长期迭代成本更低
- Vue Flow 足以承载当前阶段所需的树关系可视化与基础交互
- Mermaid 与“工作流源码存库、文本化描述流程”的设计直接对应
- Pinia 适合承载不入库的本地 UI 状态
- Vue 生态足以承接维护页、图形页、节点详情面板等多类页面

#### 4.2.1 Mermaid 的职责边界

当前版本中，Mermaid 的定位是“工作流结构展示组件”，而不是复杂交互画布。

当前明确支持的能力：

- 根据数据库中的 `mermaid_source` 渲染工作流结构
- 支持点击 Mermaid 节点
- 点击后向前端返回对应的 `mermaid_node_id`
- 前端根据当前选中的 `mermaid_node_id` 更新右侧面板内容

当前不作为核心要求的能力：

- 不要求 Mermaid 图内承担复杂业务操作
- 不要求 Mermaid 图内必须实现选中高亮
- 如果后续补充图内选中样式，也仅作为增强项，而不是当前方案前提

### 4.3 本地服务技术栈

- Web 框架：`Fastify`
- 请求与响应校验：`Zod`
- 配置校验：`Zod`
- 本地配置文件：`JSONC`
- 数据访问风格：`轻量 SQL + 少量仓储封装`

选择原因：

- Fastify 足够轻，适合本地服务，不会引入过重框架成本
- Zod 适合统一做 API 入参、出参与配置结构校验
- 当前项目的结构化数据边界清晰，适合明确 API 契约而不是隐式约定
- 数据库层需要精确控制 SQLite、CSV 同步、路径规则与特殊删除流程，不适合过重 ORM

#### 4.3.1 Fastify 的职责边界

这里的 `Web 框架` 指的是“本地 HTTP API 服务框架”，不是前端页面框架。

Fastify 当前承担的职责：

- 提供本地 HTTP API 入口
- 接收前端请求并完成参数解析
- 调用服务层模块
- 返回统一的 JSON 结果与错误语义

Fastify 当前不承担的职责：

- 不承载前端页面业务状态
- 不替代服务层业务规则判断
- 不直接决定数据库表语义、路径规则或删除保护策略

当前补充结论：

- Web API 在 `L4` 中正式采用“资源 CRUD + 显式动作接口”的混合路由风格
- 资源型能力优先按资源建模
- 业务命令型能力采用显式动作接口，并统一使用 `POST`

#### 4.3.2 Zod 的职责边界

当前版本中，Zod 主要承担“结构校验”，而不是“业务规则校验”。

Zod 负责：

- API 请求结构校验
- API 响应结构校验
- `local.config.jsonc` 配置结构校验
- 基础枚举、字段类型、必填项、字符串非空等约束

Zod 不负责：

- 判断数据库中的对象是否真实存在
- 判断父子关系是否合法或是否形成循环
- 判断删除节点时是否需要二次提醒
- 判断 `tool_key` 是否在业务上下文中允许使用

这些规则应由服务层模块负责。

### 4.4 数据库与存储技术

- 运行时数据库：`SQLite`
- SQLite 驱动：`better-sqlite3`
- Schema 管理方式：项目内 SQL 文件顺序执行
- 文件内容存储：工作目录下的结构化文件夹
- 同步格式：`CSV` + `manifest.json`

选择原因：

- SQLite 已经被系统设计明确为运行时结构化数据源
- `better-sqlite3` 适合本地单机、同步调用、低复杂度场景
- 直接维护 SQL 更贴近已确认的数据表设计与导入导出规则
- CSV 一表一文件，符合现有同步设计，也更利于 Git 管理

### 4.5 工程组织方式

推荐采用单仓多包结构：

- `web`：网站前端
- `service`：本地服务
- `shared`：共享 schema、DTO、错误码、常量

选择原因：

- 当前版本虽然规模不大，但已经天然分成前端、服务、共享边界
- 共享层可以减少接口漂移
- 后续如果增加插件壳层，只需复用 `service` 与 `shared`

## 5. 当前不采用的方案

### 5.1 不采用纯前端直连 SQLite / 文件系统

原因：

- 违背“本地服务是唯一合法读写入口”的系统约束
- 会导致业务规则、路径规则、删除保护和工具调用分散到前端
- 后续多壳层扩展时难以复用

### 5.2 不采用 Electron / Tauri 作为当前版本基础

原因：

- 当前正式设计已收敛为网站前端，不是桌面壳体优先
- 现在引入桌面壳会让问题边界变大
- 本地服务 + 网站前端已足够支撑当前需求

### 5.3 不采用 NestJS

原因：

- 当前项目是本地单机服务，不需要更重的框架分层与装饰器体系
- Fastify 更直接，更适合作为 v1 稳定落地底座

### 5.4 前端主框架统一采用 Vue 3

原因：

- 当前图形区域定位是“导航视图”，不是复杂图编辑器
- 项目的主要复杂度在节点详情、文件列表、工作流绑定、状态与记录操作等业务面板
- 主要维护者更熟悉 Vue 3，选择熟悉栈更有利于后续长期维护
- 现有树关系可视化需求并不要求引入另一套前端主框架，统一采用 Vue 技术栈更有利于长期维护

### 5.5 不采用全 ORM

原因：

- 当前数据库不是唯一事实源，且存在文件系统与 CSV 同步边界
- 表结构、路径、删除流程、同步导入导出都需要精细控制
- 全 ORM 容易让数据库访问抽象过重，不利于调试和边界控制

### 5.6 不采用 tRPC 作为前后端通信主方案

原因：

- 当前系统存在明确的“本地服务边界”
- HTTP JSON API 更利于未来接入其他壳层
- 与共享 Zod schema 组合后，类型与边界都能兼顾

---

## 6. 选型带来的约束与代价

当前技术选型带来的约束如下：

- 需要同时维护前端与本地服务两个运行单元
- 前后端联调要处理端口、启动顺序与本地配置
- SQLite + 文件系统双事实边界要求接口设计非常明确
- 轻量 SQL 虽然可控，但需要团队主动维护 SQL 与仓储边界
- 手动同步虽然安全，但用户需要理解导入导出的触发时机

这些代价是当前版本为换取“边界清晰、可控、可维护”所接受的复杂度。

---

## 7. 推荐的落地边界

基于当前选型，后续实现阶段建议优先顺序如下：

1. 工作目录与路径规则
2. SQLite 初始化与 schema 执行
3. 本地配置读取与校验
4. 文件访问基础层
5. Prompt / Workflow / Project / Node 核心领域模块
6. 节点删除保护、工具调用、同步、巡检
7. Web API
8. 网站前端页面与图形交互

---

## 8. 当前版本的暂定结论

当前版本暂定采用以下组合：

- `TypeScript`
- `Node.js`
- `Fastify`
- `Zod`
- `better-sqlite3`
- `SQLite`
- `Vue 3`
- `Vite`
- `Vue Router`
- `Pinia`
- `@vue-flow/core`
- `mermaid`
- `JSONC`
- `CSV + manifest.json`

这组技术的目标不是“技术上最炫”，而是“与既定系统设计最一致、最容易稳定落地”。

---

## 9. 待继续讨论的问题

虽然总体技术路线已经收敛，但以下问题仍建议在实现前继续确认：

- 前端是否从现在开始就拆成 `web` 独立包
- 服务是否从一开始就提供静态资源托管能力
- Mermaid 编辑器是先做源码输入框，还是一开始就接更强编辑体验
- 巡检与同步页面在 v1 中做到什么深度
- 测试优先覆盖服务层，还是前后端一起建立基础测试框架
