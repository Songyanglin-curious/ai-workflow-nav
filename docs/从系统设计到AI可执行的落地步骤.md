# 从系统设计到AI可执行的落地步骤

## 1. 文档定位

本文档不是某一份设计稿的附属说明，而是一条可以反复复用的工作流。

它要回答的问题是：

- 当我要从零开始做一个可维护系统时，应该按什么顺序推进
- 如何避免一开始就陷入技术细节，导致系统后期难维护、难扩展
- 如何把“人能理解的设计”继续收敛成“AI 可以稳定执行的任务”

这份文档的目标不是继续扩大全局抽象，而是给出一条从系统设计走到 AI 可执行的稳定路径，并让这条路径本身可以成为未来复用的工作流。

---

## 2. 适用范围

本文档适用于以下场景：

- 新系统从 0 到 1 的建设
- 已有系统准备重构，但希望先补齐设计依据
- 希望把长期开发过程沉淀成可复用工作流
- 希望未来把部分实现工作稳定交给 AI 执行

本文档不直接代替以下具体文档：

- 系统设计文档
- 模块设计文档
- 技术路线与技术选型文档
- 数据库表结构文档
- API 契约文档
- 单个任务的 AI 执行卡

它定义的是这些文档之间的产出顺序和收敛路径。

---

## 3. 核心原则

### 3.1 先定边界，再谈实现

可维护系统的第一目标不是“尽快写出能跑的代码”，而是先定清楚：

- 系统解决什么问题
- 不解决什么问题
- 哪些规则长期稳定
- 哪些部分以后允许变化

### 3.2 先有系统设计，再有技术选型

技术选型不能脱离设计依据单独先行。

因为如果连系统的核心抽象、分层结构、事实源划分都还没有定下来，就无法判断：

- 需要什么运行形态
- 需要什么存储模型
- 哪些技术是必须的
- 哪些技术只是实现偏好

所以更合理的顺序不是“先想技术，再做设计”，而是：

- 先定义问题边界
- 再做系统设计
- 再做模块拆分
- 最后基于这些结果做技术路线与技术选型

### 3.3 模块拆分也必须早于技术选型

技术选型不仅要承接系统设计，也要承接模块边界。

因为只有模块拆出来之后，才能判断：

- 哪些能力需要本地服务承接
- 哪些能力只是文件访问基础能力
- 哪些能力属于前端交互层
- 不同模块各自需要什么技术支撑

如果模块还没拆，技术路线就容易变成泛泛而谈，最后落不到真正的实现边界上。

### 3.4 先稳定抽象，再稳定底座，再做业务功能

从零开始做系统时，最先稳定下来的不应该是页面和技术细节，而应该是：

- 问题边界
- 核心抽象
- 分层结构
- 模块边界

在这些清楚之后，再去稳定：

- 运行环境
- 目录规则
- 存储规则
- 初始化流程
- 自检与修复机制
- 日志与错误语义

这些决定系统后续能不能长期维护。

### 3.5 每一层只解决本层问题

从抽象到落地必须逐层收敛，不能混层。

例如：

- 系统设计层不定义每个接口字段
- 模块设计层不直接写单次 AI 提示词
- 技术选型层不替代接口契约层
- AI 执行层不再做关键架构决策

### 3.6 AI 只负责执行，不负责补关键设计

当任务交给 AI 时，AI 最适合做的是：

- 按约束实现
- 按任务补代码
- 按验收标准自检

AI 不适合在缺少关键前提时替你决定：

- 是否应该用 SQLite 还是别的数据库
- 文件系统是不是事实源
- 删除流程要不要二次确认
- 哪些状态入库，哪些状态只放缓存

这些必须在 AI 执行前定死。

## 4. 推荐的七层收敛模型

从零开始到 AI 可执行，建议采用以下 7 层递进：

1. `L0` 问题定义与目标边界
2. `L1` 系统设计
3. `L2` 模块设计
4. `L3` 技术路线与技术选型
5. `L4` 接口与数据契约
6. `L5` 代码文件组织功能明确与文件夹和文件组织
7. `L6` 任务规格单
8. `L7` AI 执行提示包

其中，`技术路线与技术选型` 必须放在 `系统设计` 和 `模块设计` 之后。
因为技术选型必须有设计依据，而不是反过来驱动核心抽象。

---

## 5. 七层的职责与产出

### 5.1 `L0` 问题定义与目标边界

这一层回答：

- 这个系统到底要解决什么问题
- 目标用户是谁
- 最小可用范围是什么
- 当前版本不做什么
- 哪些是长期稳定目标，哪些只是阶段性实现

这一层至少应产出：

- 项目目标
- 核心使用场景
- 非目标列表
- 关键约束列表
- 版本边界

如果这一层不清楚，后面所有设计都会漂移。

### 5.2 `L1` 系统设计

这一层回答：

- 系统由哪些核心抽象构成
- 分层结构是什么
- 数据源如何分工
- 状态边界在哪里
- 同步策略是什么
- 哪些规则全局稳定

这一层解决的是：

- 系统是什么
- 为什么这么分层
- 为什么这样划分事实源

这一层不解决：

- 每个模块如何实现
- 用什么具体技术承载
- API 字段如何命名
- 一个任务拆成几步

### 5.3 `L2` 模块设计

这一层要把系统拆成稳定模块，并明确：

- 每个模块负责什么
- 每个模块不负责什么
- 模块依赖顺序是什么
- 哪些是基础设施模块
- 哪些是业务模块
- 哪些是交互模块

对于从零开始的系统，这一层尤其重要，因为它决定后续技术路线和任务拆分是否稳定。

建议优先区分三类模块：

- 基础设施模块
- 业务模块
- 交互模块

并优先实现基础设施模块。

### 5.4 `L3` 技术路线与技术选型

这一层不是凭空决定技术，而是承接前两层的结果。

这一层回答：

- 为了承载既定系统设计，需要什么技术路线
- 为了承载既定模块边界，各模块分别适合用什么技术
- 当前版本采用什么技术组合
- 为什么这些技术最匹配当前设计，而不是别的
- 当前技术选型带来的约束和代价是什么

这一层至少应定死以下内容：

- 运行形态
- 主语言与运行时
- 前端框架
- 后端服务形态
- 数据库存储方案
- 文件存储方案
- 图形或流程可视化方案
- 配置文件格式
- 测试方式
- 日志与错误处理方式
- 构建与启动方式

这一层的关键不是“列技术名词”，而是说明技术与设计之间的对应关系。

对于当前项目，这一层至少应明确：

- 因为当前版本以前端网站作为唯一交互入口，所以前端技术围绕网站形态选择
- 因为业务合法读写入口是本地服务，所以需要本地服务承接统一读写
- 因为运行时需要元数据、关系和查询能力，所以使用 SQLite
- 因为正文和真实内容要落盘，所以文件系统承载正文内容
- 因为工作流需要文本化描述，所以使用 Mermaid
- 因为树和节点关系需要可视化交互，所以使用 Vue Flow

### 5.5 `L4` 接口与数据契约

这一层是 AI 落地前最关键的一层。

需要定死的内容包括：

- API 路径与方法
- 请求参数与返回结构
- 错误语义
- 数据库字段语义
- 目录结构契约
- 文件命名规则
- 初始化规则
- 自检规则
- 自修复规则
- 导入导出格式

每个契约至少应明确：

- 输入是什么
- 输出是什么
- 成功条件是什么
- 失败条件是什么
- 是否写数据库
- 是否写文件
- 是否需要确认
- 是否允许自动修复

如果缺少这一层，AI 最容易乱猜。

### 5.6 `L5` 任务规格单

这一层要把模块继续拆成 AI 或人类工程师都能一次完成的小任务。

每个任务规格单建议包含：

- 任务标题
- 背景
- 目标
- 范围
- 非目标
- 涉及模块
- 修改范围
- 验收标准
- 风险与注意事项

任务粒度建议如下：

- 一个任务只做一个清晰能力
- 一个任务尽量只跨一小段依赖链
- 一个任务完成后应能独立验证

不推荐的标题：

- 完成后端功能
- 完成节点模块

推荐的标题：

- 实现 schema 文件扫描与顺序执行器
- 实现工作目录自检与自动建目录
- 实现 projects 读取、创建、更新 API 与 project 删除保护动作
- 实现 project_nodes 删除前保护流程

### 5.7 `L6` AI 执行提示包

这一层是最终交给 AI 的执行材料。

每个 AI 执行提示包建议固定包含：

- 背景
- 目标
- 已知约束
- 依赖前提
- 需要修改的文件范围
- 明确禁止事项
- 验收标准
- 测试或人工检查方式

到了这一层，AI 应该做到：

- 不需要再猜系统目标
- 不需要再猜系统设计
- 不需要再猜模块边界
- 不需要再猜技术路线
- 不需要再猜接口结构
- 不需要再猜目录规则
- 不需要再做关键架构决策

## 6. 从零开始的推荐推进顺序

如果是从 0 到 1 做一个可维护系统，推荐按下面顺序推进。

### 6.1 先做 `L0`

先定义：

- 目标
- 约束
- 非目标
- 当前版本边界

这一阶段最怕的问题是目标膨胀。

### 6.2 再做 `L1`

先做系统设计，而不是先定技术。

这一阶段至少要回答：

- 系统有哪些核心抽象
- 哪些数据是事实源
- 整体如何分层
- 状态边界在哪里
- 当前版本的全局规则是什么

如果这一层没做，技术选型就没有依据。

### 6.3 再做 `L2`

在系统设计之后，继续做模块拆分。

这一阶段至少要回答：

- 系统应拆成哪些稳定模块
- 每个模块负责什么
- 每个模块不负责什么
- 模块之间怎样依赖
- 哪些模块必须优先落地

如果这一层没做，技术路线和任务拆分都会失去稳定边界。

### 6.4 再做 `L3`

在系统设计和模块拆分都清楚之后，再做技术路线与技术选型。

这一阶段至少要回答：

- 哪种运行形态最适合承载当前设计
- 哪组技术最适合承载当前模块边界
- 各个模块各自依赖什么技术能力
- 当前版本先采用什么，后续哪些再演进

这时技术选型才是“有依据的承接”，而不是拍脑袋决定。

### 6.5 再做 `L4`

把所有关键接口和规则定死。

对于可维护系统，至少应优先补齐三类契约：

1. API 契约
2. 文件与目录契约
3. 初始化 / 自检 / 修复契约

### 6.6 再做 `L5`

把模块拆成可以逐个完成、逐个验证的任务规格单。

这一步完成后，项目才真正进入稳定执行阶段。

### 6.7 最后做 `L6`

把任务规格单转成 AI 执行提示包。

只有到这一步，AI 才适合直接进入实现。

在真正进入实现时，推荐优先落地的通常是：

1. 启动自检与环境引导
2. 数据库初始化与 schema 执行
3. 工作目录与路径规则
4. 配置读取与校验
5. 文件访问基础能力

这些属于“先稳定底座，再展开业务”的落地顺序。

## 7. 可维护系统视角下，最容易漏掉的内容

从零开始做系统时，最容易被忽略，但实际上必须尽早明确的内容包括：

### 7.1 初始化与自检

至少要明确：

- 首次启动要检查什么
- 缺失目录是否自动创建
- 缺失数据库是否自动创建
- schema 执行失败怎么处理
- 自检失败是中止还是降级

### 7.2 路径与命名规则

至少要明确：

- 相对路径与绝对路径的边界
- 文件名合法字符规则
- 固定目录结构
- 是否允许移动和改名

### 7.3 事实源划分

至少要明确：

- 哪些数据以数据库为准
- 哪些内容以文件系统为准
- 同步阶段以什么为准

### 7.4 删除与危险操作保护

至少要明确：

- 哪些操作需要确认
- 哪些操作需要二次确认
- 哪些操作禁止自动执行

### 7.5 可测试性

至少要明确：

- 每一层如何验证
- 哪些可以自动测试
- 哪些只能人工验收
- AI 完成后如何检查结果

如果这些内容缺失，系统即使写出来，也很难长期维护。

## 8. 判断是否已经达到“AI 可直接执行”

当以下条件基本满足时，说明已经可以把任务稳定交给 AI：

1. AI 不需要再猜系统目标和边界。
2. AI 不需要再猜系统的分层规则。
3. AI 不需要再猜模块职责和依赖关系。
4. AI 不需要再猜技术栈和运行方式。
5. AI 不需要再猜接口参数、返回结构和错误语义。
6. AI 不需要再猜目录结构、命名规则和路径行为。
7. AI 不需要再猜哪些内容写数据库、哪些内容写文件。
8. AI 不需要再猜什么算完成，可以直接对照验收标准执行。

如果上述条件还不满足，就说明任务还停留在设计阶段，而不是执行阶段。

## 10. 一句话总结

从零开始做一个可维护系统，正确路径不是“想到什么就实现什么”，而是按下面顺序逐层收敛：

`问题定义 -> 系统设计 -> 模块设计 -> 技术路线与技术选型 -> 接口与数据契约 -> 任务规格单 -> AI 执行提示包`

其中，`技术路线与技术选型` 不能脱离设计依据单独提前。
只有先有系统设计和模块拆分，技术选型才有依据；只有这些都收敛清楚之后，AI 才能稳定执行，而不是边做边替你做设计决策。
