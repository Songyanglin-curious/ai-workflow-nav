# 个人 AI 工作流导航器系统设计 v1.1

## 1. 文档定位

本文档是当前阶段的正式设计整理稿，用于统一项目的核心概念、数据边界、存储方式、同步方式、页面职责和实现约束。

本文档的目标不是描述所有实现细节，而是给后续数据库设计、接口设计、前端页面和同步机制提供稳定约束。

当前文档结论以“已确认设计”为准，不再保留讨论过程中的中间表述。

---

## 2. 项目目标

本项目是一个面向个人使用的 AI 工作流导航器，核心目标如下：

1. 管理提示词、工作流、项目、节点、方案等多层对象。
2. 让项目节点承载实际工作上下文，并关联工作流、对话记录和总结记录。
3. 通过本地服务统一管理业务数据，避免多入口直接改文件或数据库。
4. 用 SQLite 支撑运行时元数据、关系和查询能力。
5. 用磁盘文件承载真实内容，并通过同步导出文件实现跨机器同步。
6. 兼容未来多个交互壳层，例如浏览器插件、Trae 插件、独立页面。

---

## 3. 核心设计原则

### 3.1 单一合法入口

所有业务数据的合法读写入口都应是本地服务。  
浏览器插件、未来的 Trae 插件、独立页面都只是交互壳层，不能直接绕过服务层操作数据库或磁盘。

### 3.2 分层调用

系统采用三层架构：

1. 上层交互层
2. 中层服务层
3. 底层文件层

约束如下：

- 只能逐层调用
- 不能跨层直写

### 3.3 ID 优先

所有领域对象都必须有唯一 ID，所有真实关联都统一使用 ID。  
名称仅用于显示、检索和人工理解，不参与真实关联。

### 3.4 运行时与同步时分离

运行时和同步时的数据源职责不同：

- 运行时，SQLite 是元数据、关系和查询能力的唯一数据源。
- 文件类内容本身始终以磁盘中的实际文件为准。
- 同步阶段，导出的文本文件是同步过程中的唯一数据源。
- 从 Git 拉取后的恢复过程，以同步文件和磁盘文件作为恢复输入源。

### 3.5 手动同步

同步只允许手动触发，不做自动同步、自动覆盖或自动回写。  
导入导出何时执行，由人决定。

## 4. 基础规则

### 4.1 文件命名规则

允许字符：

- 中文
- 英文
- 数字
- 下划线 `_`
- 短横线 `-`
- 空格
- 普通括号
- 点号 `.`

禁止字符：

- `<`
- `>`
- `:`
- `"`
- `/`
- `\`
- `|`
- `?`
- `*`

额外限制：

- 文件名不能以空格结尾
- 文件名不能以 `.` 结尾
- 禁用保留名：`CON`、`PRN`、`AUX`、`NUL`、`COM1~COM9`、`LPT1~LPT9`

### 4.2 唯一 ID 规则

- 所有领域对象都必须有唯一 ID
- 全局唯一 ID 统一使用 `UUIDv7`

### 4.3 路径规则

- 数据库中涉及文件路径和文件夹路径的字段，统一存相对路径
- 不在业务数据层存绝对路径
- 绝对路径只在本地配置和实际执行阶段出现

## 5. 数据源模型

### 5.1 运行时数据源

运行时由 SQLite 承担以下职责：

- 元数据
- 关系
- 查询
- 索引
- 页面展示所需的结构化数据

### 5.2 文件内容源

以下内容以磁盘文件为准：

- 提示词正文
- 节点对话记录文件
- 节点总结文件
- 其他真实落盘内容

### 5.3 同步数据源

同步时采用 `dbSyncs/` 目录下的导出文件作为同步输入输出。  
当前已确认采用“多文件 CSV，一表一文件”的导出方式，并保留一个 `manifest.json` 作为全局说明文件。

## 6. 领域模型

### 6.1 Prompt

用途：管理提示词及其元数据。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `content_file_path`
- `created_at`
- `updated_at`

规则：

- 一份提示词对应一个 `.md` 文件
- 正文存文件
- 元数据和检索信息存 SQLite

### 6.2 Workflow

用途：描述可复用的工作流对象。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `mermaid_source`
- `created_at`
- `updated_at`

规则：

- `mermaid_source` 存整段 Mermaid 源码文本
- 当前版本工作流不单独建立落盘内容目录
- 工作流结构、元数据和关系由数据库承载

### 6.3 WorkflowNodeActionBinding

用途：将工作流中的 Mermaid 节点绑定到一个可执行动作目标。

建议字段：

- `workflow_id`
- `mermaid_node_id`
- `action_type`
- `target_ref`
- `created_at`
- `updated_at`

动作类型当前约定：

- `prompt`
- `tool`
- `link`

规则：

- 一个 Mermaid 节点当前最多绑定一个主动作
- 节点允许不绑定动作
- `action_type = prompt` 时，`target_ref` 指向 `Prompt.id`
- `action_type = tool` 时，`target_ref` 指向本地工具配置中的 `tool_key`
- 保存绑定关系时，必须基于当前 Mermaid 节点集合做差异比对并清理失效绑定

### 6.4 Project

用途：承载一组项目节点及其关系。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `created_at`
- `updated_at`

### 6.5 ProjectNode

用途：承载实际工作上下文，是系统的核心使用单位。

建议字段：

- `id`
- `name`
- `description`
- `status`
- `created_at`
- `updated_at`

规则：

- 项目节点不允许跨项目复用
- 项目节点关联工作流时当前限定为一对一

### 6.6 ProjectNodeRelation

用途：表达项目中的父子结构关系。

建议字段：

- `project_id`
- `parent_project_node_id`
- `child_project_node_id`
- `sort_order`

规则：

- 一条边一行
- 项目结构关系只表达结构，不混入运行时字段

### 6.7 ProjectNodeWorkflowBinding

用途：绑定项目节点与工作流。

建议字段：

- `project_node_id`
- `workflow_id`
- `created_at`
- `updated_at`

### 6.8 Solution

用途：作为项目的集合管理层。

建议字段：

- `id`
- `name`
- `description`
- `tags`
- `category`
- `created_at`
- `updated_at`

### 6.9 SolutionProjectBinding

用途：表达方案与项目之间的集合关系。

建议字段：

- `solution_id`
- `project_id`
- `sort_order`

规则：

- 方案与项目的关联统一通过 `project_id`
- 一个项目可以属于多个方案

### 6.10 ConversationRecord

用途：挂载到节点上的对话记录目录入口。

建议字段：

- `id`
- `project_node_id`
- `folder_path`
- `created_at`
- `updated_at`

规则：

- `folder_path` 存相对路径
- 一个节点的对话记录以多个 `.md` 文件形式存放在该目录中
- 默认追加到最新对话文件
- 支持手动新建新的对话记录文件

### 6.11 InsightRecord

用途：挂载到节点上的总结/认知目录入口。

建议字段：

- `id`
- `project_node_id`
- `folder_path`
- `record_type`
- `created_at`
- `updated_at`

规则：

- `folder_path` 存相对路径
- 总结与认知内容由目录下文件列表表达

### 6.12 Config

用途：存储本地配置。

建议内容：

- 数据根路径
- 本地服务端口
- 外部工具配置
- 模型 Key
- 其他本地运行参数

## 7. 状态模型

### 7.1 持久状态

节点的长期业务状态采用：

- `default`
- `todo`
- `fix`

规则：

- 一个节点同一时刻只有一个持久状态
- 持久状态由人手动判断和切换

### 7.2 运行时状态

运行时状态不进入节点主表，单独维护。

当前约定：

- `active_project_node_id`：当前选中的项目节点
- `active_workflow_node_id`：当前位于工作流的哪个节点

规则：

- 允许为空
- 与当前项目或当前视图上下文关联
- 不与节点长期状态混用

### 7.3 UI 状态边界

以下属于 UI 状态，不进入业务主表：

- 当前展开面板
- 悬浮窗是否折叠
- 搜索关键词
- 当前缩放比例
- 其他仅服务于当前界面的临时状态

## 8. 功能设计

### 8.1 提示词管理

支持：

- 增删改查
- 快速复制正文

### 8.2 工作流管理

支持：

- 增删改查
- Mermaid 编辑与渲染
- 节点动作绑定维护

### 8.3 项目与节点管理

支持：

- 项目的增删改查
- 节点的增删改查
- 项目结构关系维护
- 节点与工作流绑定

### 8.4 节点相关能力

节点需要支持：

- 展示名称与描述
- 展示工作流名称
- 展示节点状态
- 展示当前位于工作流的哪个节点
- 触发工作流节点绑定动作
- 当节点动作类型为 `prompt` 时快速复制提示词
- 当节点动作类型为 `tool` 时调起本地工具
- 展示对话目录下的文件列表
- 展示总结目录下的文件列表
- 调用外部工具打开关联文件

### 8.5 对话记录保存

当前确认支持：

1. 将剪贴板内容追加进最新对话文件
2. 在节点上手动新建对话记录文件

前端需要提供的接口语义：

- 读取节点对话目录信息
- 读取对话目录下的文件列表
- 追加内容到最新对话文件
- 在目录下创建新的对话记录文件

### 8.6 巡检能力

系统需要支持巡检以下问题：

- 有索引但文件不存在
- 文件存在但索引缺失
- 工作流节点未绑定动作
- 项目节点未绑定工作流
- 绑定关系引用了不存在的对象
- `tool` 类型动作引用了不存在的本地工具定义

## 10. 存储设计

### 10.1 工作目录结构

当前版本采用如下工作目录结构：

```text
工作目录/
  db/
    runtime.sqlite
    template.sqlite
  dbSyncs/
    manifest.json
    prompts.csv
    workflows.csv
    workflow_node_actions.csv
    projects.csv
    project_nodes.csv
    project_node_relations.csv
    project_node_workflows.csv
    solutions.csv
    solution_projects.csv
    conversations.csv
    insights.csv
  prompts/
    prompt-name__shortId.md
  projects/
    project-name__shortId/
      node-name__shortId/
        chatLogs/
        summaries/
  local.config.jsonc
```

### 10.2 项目与节点目录命名

项目实例目录与节点实例目录采用：

- `初始名称 + "__" + shortId`

规则：

- 创建时生成
- 生成后将目录名写入数据库
- 后续即使名称修改，目录名也不变
- 真实关联仍然通过 ID 维护

### 10.3 固定子目录

节点目录下固定子目录当前采用：

- `chatLogs`
- `summaries`

说明：

- `summarys` 已正式修正为 `summaries`

### 10.4 当前为什么没有 `workflows/` 目录

当前版本不单独建立 `workflows/` 目录。

原因：

- 工作流的结构、关系和 `mermaid_source` 存在数据库中
- 同步时通过 `dbSyncs/workflows.csv` 等表级文件导出
- 当前工作流没有独立的正文型落盘目录需求

### 10.5 文件映射基线

当前版本已确认采用以下映射基线：

1. 数据库中的文件路径和文件夹路径统一存相对路径
2. 真实文件内容按目录约定落盘
3. 不同类型内容的落点必须稳定且可预测
4. `dbSyncs/` 采用“一个表一个文件”的多文件导出方案
5. `manifest.json` 负责导出版本和全局说明

扫描恢复原则：

- 目录约定负责定位真实文件内容
- `dbSyncs/` 负责恢复结构化元数据与关系
- 运行时仍以 SQLite 为唯一结构化数据源

## 11. 同步设计

### 11.1 导出格式

当前已确认：

- 采用多文件导出
- 采用一表一文件
- 表级文件采用 `CSV`
- 全局说明文件采用 `manifest.json`

### 11.2 导入导出策略

当前已确认：

- 由人决定何时导入
- 由人决定何时导出
- 导入时允许先清空再重建
- 导入失败时不做自动回滚
- 出错后由人排错并重新导入
- 页面提供刷新按钮，导入后由人手动刷新

### 11.3 同步冲突的真实含义

这里的“冲突”主要不是 Git merge conflict，而是数据源切换时的状态不一致，例如：

- 磁盘上的同步文件已更新，但运行中的 SQLite 还是旧数据
- 刚导出同步文件后，又继续修改了运行时数据
- 导入同步文件时，旧索引或旧关系尚未清理

当前策略是：

- 不追求自动回滚
- 优先保证流程简单、行为可控、人工可理解

## 12. 外部工具配置设计

### 12.1 配置模型

外部工具配置采用 `JSONC`，当前建议整合在 `local.config.jsonc` 中的对应配置段内。

逻辑结构采用：

- `version`
- `defaults`
- `tools`
- `routes`
- `security`

### 12.2 工具定义

`tools` 用于定义可执行工具，建议字段包含：

- `tool_key`
- `label`
- `enabled`
- `platform`
- `exePath`
- `actions`

### 12.3 动作定义

`actions` 当前建议支持：

- `openFile`
- `openFolder`
- `openPath`
- `openAtLine`

每个 action 建议包含：

- `args`
- `requiresExists`

占位符约定：

- `{filePath}`
- `{folderPath}`
- `{path}`
- `{line}`
- `{column}`

### 12.4 路由与业务绑定边界

- 业务层已经通过 `tool_key` 指定工具时，优先使用业务绑定
- `routes` 只用于默认兜底，不替代表达业务语义
- 当前版本 `routes.match` 先只支持按扩展名匹配

### 12.5 安全要求

- `allowAbsolutePaths` 默认建议为 `false`
- 业务数据层继续只传相对路径
- 服务层在执行前解析为绝对路径
- 所有目标路径都必须经过规范化和允许根目录校验
- `exePath` 允许为本机绝对路径

## 13. 排序、删除与结构约束

### 13.1 排序字段

当前统一采用：

- `sort_order INTEGER`

规则：

- 数值越小越靠前
- 只在同一父级范围内有意义
- 不做全局排序
- 默认可按较大步长分配，例如 `1000`、`2000`、`3000`

适用范围：

- 同一父节点下的子节点顺序
- 同一方案下的项目顺序
- 其他需要人工排序的同级对象

### 13.2 删除与重命名

当前确认：

- 删除采用级联删除
- 重命名不影响关系

原因：

- 所有真实关联都走 ID，不走名称

### 13.3 项目节点复用约束

当前确认：

- 项目节点不允许跨项目复用

## 14. 架构设计

### 14.1 总体架构

项目整体架构为：

上层交互层  
↓  
中层服务层  
↓  
底层文件层

### 14.2 上层交互层职责

上层只表达“我要做什么”，例如：

- 新建节点
- 关联工作流
- 保存对话
- 读取节点详情

上层不直接接触：

- 文件
- 目录结构
- 路径拼接

### 14.3 中层服务层职责

中层是唯一调度层，负责：

- 生成 ID
- 更新索引
- 维护关系
- 处理排序
- 决定文件写入顺序
- 调用扫描器
- 返回统一数据结构

中层不负责：

- 页面展示
- 浏览器 DOM 提取
- 任意自由拼接正文格式

### 14.4 底层文件层职责

底层只负责：

- 读文件
- 写文件
- 列目录
- 改名
- 删除
- 判断文件是否存在

底层不理解：

- `node` 是什么
- `workflow` 是什么
- 关系是否合理

## 16. 当前正式结论

截至本文档版本，当前最关键的正式结论如下：

1. 运行时由 SQLite 承担元数据、关系和查询能力的唯一数据源职责。
2. 文件类内容始终以磁盘中的实际文件为准。
3. 同步阶段由 `dbSyncs/` 中的导出文件承担唯一同步数据源职责。
4. 所有真实关系统一使用唯一 ID。
5. 节点长期状态与运行时状态必须拆开管理。
6. 项目和节点目录采用“初始名称 + `__` + shortId”命名并在创建后保持稳定。
7. 当前版本采用“多文件 CSV，一表一文件”的同步导出方案。
8. 外部工具配置采用 `JSONC` 结构化配置，业务层只保存 `tool_key` 引用。

这些结论已经足够支撑后续数据库表结构、接口设计和前端实现，不再属于待讨论事项。
