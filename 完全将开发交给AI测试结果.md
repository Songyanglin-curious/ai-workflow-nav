
# 背景

2026年4月13日，设计的个人AI工作流导航器，已做好：

* 系统设计（人主导）
* 模块拆分（人主导）
* 技术选型（人主导）
* 数据库表设计（人审核）
* 接口与数据契约（人审核）
* 代码文件组织（人审核）

整个设计成稿，15万字左右。

让AI拆分了65个小任务，使用 codex 逐个任务执行。

当所有任务执行完成后，代码无报错，但是给出的结果是无法被使用的。

# 结果

1. 哪怕拆分到文件层级，AI仍然无法交付出一个业务正确、结构坚固、可维护的成果。
2. 也许让AI完成，需要做到代码层级的约束；但是，做到代码层级的约束，设计的难度比直接写代码的难度高出许多。这就陷入了悖论。
3. 目前AI可以做掉大部分实现，但是最终产物要做到严格不跑偏、完全贴合初始设计、而人还几乎不碰实现层，目前还不行。

# 总结

1. 未来AI的实现方面，使用它的范围需要限制在非常细小的边界，不能一次修改太多模块与代码，不能超出我的审核带宽。
   而且，要让它局限于模块内聚、易测试的范围。
2. AI进行问题探讨阶段，AI做方案探讨还是很不错的；但是需要注意的点在于：

   * AI很难跳出最开始的限定框架；
   * AI会非常容易顺着人的方向说，就像一个溜须拍马的、部分聪明部分蠢的人。

   可以同时使用多个AI、隔离上下文，来讨论同一个问题，会有惊喜。
3. AI写的文档，普遍性问题在于：结构完整，但是信息密度低，看的很累。
   需要不断地调试，但是不能让AI来重复迭代；迭代出来的结果就是：有文档，但是人看不进去。
4. 核心原则是：AI不稳定，任何AI的交互以人为主导。

   * 讨论是平等的，做决定以人为主；
   * 学习要主动，AI是非常好的学习助手；
   * 我思故我在，脑子不能交给AI。
